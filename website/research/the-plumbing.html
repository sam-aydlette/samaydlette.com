The Plumbing: A Composable Reference Implementation for FedRAMP 20x and Rev 5

Two frameworks, one truth, and the layer below them.

Author's Note

This paper documents an implementation that runs on the infrastructure of this website. Every assertion the paper makes about the artifacts it describes is verifiable: anyone can curl the published documents from /.well-known/, run cosign verify-blob against the Sigstore bundle, browse the artifacts in the live trust dashboard, and check that what the paper claims matches what is actually being served. The implementation is open-source at github.com/sam-aydlette/samaydlette.com.

The system is small. One CSP, one operator, no end users, no federal customer data. The patterns it demonstrates are not. The paper is explicit about what the implementation proves end-to-end and the few specific places where the claim is demonstrated by schema construction rather than by a working multi-party federation.

I wrote earlier in Pearls, PURLs and the FedRAMP 20x Inventory Problem about why a standardized component-naming layer is what FedRAMP 20x needs to deliver on its premise. This paper is the implementation of that argument, plus the part the article didn't cover: how the same primitive serves Rev 5 SSPs, and what the broader business and societal benefits are when compliance reports start composing.

Abstract

Compliance evidence in federal cloud is structurally fragmented. CSPs produce per-system reports (KSI signals under FedRAMP 20x, SSPs under Rev 5, vulnerability scans, SBOMs, audit trails), each in its own format, with its own identifiers, requiring its own reconciliation when consumers want a portfolio view. The fragmentation is the cost: assessment time, drift between SSPs and live systems, the impossibility of cross-CSP risk aggregation, the assessment-to-authorization gap that produces "compliance as a snapshot" instead of "security as a continuous property."

This paper presents a reference implementation that addresses the fragmentation by treating the naming of components as a primitive layer below all the reports. Components are named in canonical form: Package URL for software, native cloud-resource ID paired with a normalized type, content hash for static artifacts, HBOM reference for hardware. Every report a system produces references components by these names. Two systems naming the same thing produce bit-identical identifiers, and reports compose across systems without reconciliation.

The implementation runs on a static personal website. It includes: a JSON Schema for KSI signals extended with normalized component identifiers, FIPS 199 security categories, and information-flow summaries per the FedRAMP 20x Minimum Assessment Scope rule; a deploy-time emitter that generates a signal from Terraform state, package locks, content hashes, GitHub Actions provenance, and OPA policy results; a runtime emitter (AWS Lambda) that re-validates the live configuration daily; Sigstore keyless signing of the deploy-time signal with verification anchored in the public Rekor transparency log; an OSCAL Rev 5 System Security Plan generator and an OSCAL Plan of Action and Milestones generator (both validated against the official NIST OSCAL schemas) that derive from the same canonical inventory; a folder of family policy documents (one per -1 control) that the SSP references by path; a Vulnerability Detection and Response aggregator that runs at build time, classifies findings on the FedRAMP 20x PAIN scale, and blocks the deploy if any finding exceeds the Class C remediation tolerance; a Significant Change Notification gate that requires every pull request to declare its change-class; and a FedRAMP Integrated Inventory Workbook (SSP Appendix M) projector that emits the same canonical inventory in IIW column shape. Eight artifacts are published at /.well-known/ on every deploy.

The paper closes by examining what the implementation demonstrates that generalizes (the architectural pattern, the layer-below-reports framing, the dual-framework derivation, the public verifiability), and what would be strengthened by an actual cross-CSP federation. The technical pattern is implementable today. The hard work that remains is governance, not engineering.

Background: The Inventory Problem

Compliance evidence in federal cloud is produced per-system, in framework-specific formats, with framework-specific identifiers. Two FedRAMP 20x KSI signals from different CSPs nominally describe similar things (components and validations) but they do not compose into a portfolio view, because the components are named differently. CSP A identifies an EC2 instance by ARN; CSP B identifies an Azure VM by resource ID. Both are "compute instances" in any reasonable model. No automatic join exists across them.

The same fragmentation appears within a single CSP across frameworks. A 20x KSI signal and a Rev 5 SSP from the same provider describe the same system, but they are produced by different teams from different source data and rarely reconcile without manual reviewer effort. The result is what assessors call SSP drift: the security plan documents one configuration, the live system has evolved into another. Most audit findings reduce to some form of this gap.

Cross-portfolio risk aggregation is the most visible casualty. A query like "which of my authorized systems contain a particular library that just had a critical CVE" cannot be answered in any reasonable time, because the systems were inventoried by different schemas and the join across them does not exist. Each system has reasonable inventory; the inventories do not compose.

The argument I have made elsewhere is that the durable answer is to push standardization down a layer. Standardize the names of the components the reports talk about, not the reports themselves. Once components are named the same way everywhere, anything referencing them composes: SBOMs, KSI signals, SSPs, vulnerability reports. The substrate is shared; the formats can vary above it.

The Architectural Move: Layer Before Report

The reframe at the heart of this implementation is to treat the canonical inventory as a layer rather than a deliverable: the architectural primitive that makes everything above it composable.

The distinction matters because most compliance discussions blur it. The phrase "we need better inventories" is usually heard as "we need a better deliverable called Inventory." That framing concedes the structure of the problem to the existing fragmented model: another report, another reconciliation, another deliverable. The right framing is closer to "we need every report to reference components the same way." The inventory is what every report uses, not a separate document anyone reads.

The closest analog comes from networking. TCP/IP became the universal substrate of the internet by solving the addressing problem one level below where applications care, not by being the most expressive protocol. HTTP, SMTP, DNS, SSH, and every other application protocol that rides on top of TCP/IP never had to negotiate addresses with each other. They could each evolve independently, each express their own semantics, each handle their own concerns: the layer below resolved example.com to 93.184.215.14 the same way for all of them. The composability of the modern internet is a property of the addressing layer, not of any single protocol that runs on it.

The canonical inventory plays the same architectural role for compliance reports. Package URL identifies pkg:npm/express@4.17.1 the same way regardless of who is reporting it. A SHA-256 content hash identifies the bytes of a file the same way regardless of which CDN serves the file. An ARN paired with a normalized type identifies "this S3 bucket, of type object_store" the same way regardless of which framework's report is asking. Two systems naming the same thing produce bit-identical identifiers. Reports composed of those identifiers join automatically.

Architecturally, this is significant: the difference between an ecosystem of reports that fragments at every framework boundary and one that composes across them. And the move requires no single party (not FedRAMP, not NIST, not CISA, not any individual CSP) to control the namespace. PURL is an open standard, content hashes are physics, and cloud-resource ARNs are CSP-internal but globally unique by construction. The composability is a property of the identifiers themselves, not of any registry that issues them.

When this layer is in place, a KSI signal is no longer a "framework-specific report a CSP produces." It is "a structured document that references inventory components and reports validation results against them, in a 20x-shaped wrapper." An OSCAL Rev 5 System Security Plan is "a structured document that references inventory components and asserts control implementations against them, in a 53-shaped wrapper." An SBOM is the same again, in a CycloneDX or SPDX wrapper. A vulnerability report is the same. None of these reports needs to know about the others. They all reference the same components, and consumers join on that.

Implementation Overview

The implementation runs on a static website hosted on AWS: S3 for origin, CloudFront for CDN, Lambda for runtime validation, Route 53 for DNS, ACM for TLS. The compliance pipeline is a layer on top, expressed as Terraform configuration and Open Policy Agent policies. GitHub Actions drives every deploy. Deploys go through this sequence:

  1. OPA evaluates the Terraform plan and every HTML artifact against the policies in infrastructure/policies.rego. Violations block the deploy.
  2. Terraform applies the configuration.
  3. The deploy-time emitter (scripts/build-ksi-signal.py) joins five sources into one canonical document: Terraform state (cloud components), the Lambda's package-lock (npm components with PURLs), the website tree (HTML artifacts hashed), GitHub Actions environment variables (provenance), and the OPA validations from step 1.
  4. cosign signs the document via Sigstore keyless, using the GitHub Actions OIDC identity. The certificate is issued by Fulcio; the signature is recorded in the public Rekor transparency log.
  5. The OSCAL SSP generator (scripts/build-oscal-ssp.py) reads the canonical inventory and the FedRAMP KSI catalog, and emits a NIST OSCAL System Security Plan in the FedRAMP Rev 5 Moderate baseline shape (full 323-control coverage, plus eight KSI-extension controls beyond the baseline). Each in-scope NIST 800-53 control is an implemented-requirement with differentiated status, origination, and a real implementation statement. Per-family policy and procedures (the -1 controls) are satisfied by a folder of short policy documents at docs/policies/, one per family, that the SSP -1 implementation statements reference by path.
  6. The VDR aggregator (scripts/build-vdr-report.py) walks the OPA gate output (and any Checkov, tfsec, or Dependabot output that is present), classifies each finding into the FedRAMP 20x VDR evaluation framework (PAIN N1-N5, IRV, LEV, KEV), and emits a Class C-tiered report at /.well-known/vdr-report.json. The build is the report. If any finding exceeds the Class C remediation tolerance the build fails — no separate monthly reporting cycle is needed because the per-deploy emission satisfies the at-least-monthly cadence required by VDR-RPT-PER and VDR-TFR-MHR.
  7. The OSCAL POA&M generator (scripts/build-oscal-poam.py) emits a NIST OSCAL Plan of Action and Milestones document that imports the OSCAL SSP at /.well-known/oscal-ssp.json. Each POA&M item carries FedRAMP-namespaced props matching the Appendix O field set (POAM ID, Controls, Asset Identifier, Original Risk Rating, Adjusted Risk Rating, Risk Adjustment, Status, etc.). Validated against the official NIST OSCAL POA&M schema.
  8. The IIW projector (scripts/build-iiw.py) reads the canonical inventory and emits a CSV in the column shape of the FedRAMP Integrated Inventory Workbook (SSP Appendix M). The OSCAL SSP already carries the structured inventory in system-implementation/components; the IIW CSV is a flat, Excel-friendly projection of that for FedRAMP processes that expect the workbook form.
  9. All artifacts published at /.well-known/ on every deploy: the KSI signal, the Sigstore bundle, the schema, the OSCAL SSP, the OSCAL POA&M, the VDR report, the IIW CSV, and (separately, on a daily cadence from the runtime Lambda) the runtime KSI signal.

Two FedRAMP 20x rules sit alongside the deploy chain rather than inside it. Significant Change Notifications (SCN) are enforced at PR time: every pull request must carry an SCN-Type tag in its description (one of routine-recurring / adaptive / transformative, with routine-recurring as the default), validated by a separate workflow at .github/workflows/scn-tag.yml. The git history is the SCN audit record (SCN-CSO-MAR). The Secure Configuration Guide (SCG) lives at docs/policies/secure-configuration-guide.md and covers the operator's two top-level administrative accounts (AWS root, GitHub repository owner). For families inherited from AWS — PE, MA, MP, parts of CP, parts of SC — the policy docs cite the AWS authorization package AGENCYAMAZONEW as the source of authority and stop there.

The Minimum Assessment Scope (MAS) rule shows up in the schema. Every component in the canonical inventory now carries a FIPS 199 security category (confidentiality, integrity, availability) and an information-flow summary (direction, counterparty, channel, data_class). The defaults are system-specific: a public static site lands at low/moderate/low (driven by integrity), high-water mark Moderate. A different system would override the per-component defaults; the schema enforces that the fields exist and are populated, the values are an editorial decision.

Separately, an AWS Lambda function (infrastructure/lambda/index.js) runs daily on an EventBridge schedule. It reads the deploy-time KSI signal, queries the live AWS configuration of every cloud component named in the inventory, and evaluates the same Rego policies the deploy gate uses by loading them as a Wasm module compiled at build time via opa build -t wasm. The Lambda is a thin host: input transformation from AWS API responses into the resource shape the policies expect, then policy evaluation, then signal emission. There is no JavaScript port of the rules. The deploy-time gate and the runtime emitter run the identical compiled bytes; drift between them is structurally impossible. The Lambda writes a fresh runtime KSI signal at /.well-known/ksi-signal-runtime.json with current pass/fail validations and a current timestamp.

The runtime emitter exists so drift is detectable from outside. If the live S3 bucket loses encryption or the CloudFront distribution drops below TLS 1.2, the runtime signal's validations will reflect that within 24 hours, and the divergence between the deploy-time signal (which says everything was fine when last shipped) and the runtime signal (which reports current reality) is the surface a portfolio consumer reads to detect tampering.

The total annual operating cost of this compliance pipeline is approximately $125. That number is not a typo. The Lambda executions, EventBridge rules, and CloudWatch logs together come to about $125 per year for a daily cadence. The KMS, Sigstore, OPA, and AWS API costs are zero.

The Canonical Inventory in Practice

The canonical inventory is the components[] field of the KSI signal. Each entry has a normalized type drawn from a small enum, an optional native_id (the CSP's own identifier where applicable), and a global_id block carrying one or more cross-CSP identifiers. The schema is published at /.well-known/ksi-signal.schema.json.

Five normalization decisions make the inventory composable:

Software components are identified by Package URL. PURL is global by construction. A consumer querying pkg:npm/left-pad@1.3.0 across all KSI signals from all CSPs gets exactly the systems that include it, regardless of CSP. This works because PURL is a self-describing global identifier; it would not work if each CSP minted its own software identifier.

Container images are identified by OCI digest. The sha256:<hex> form serves the same role as PURL for software libraries: a global, content-addressable identifier that does not depend on which registry hosts the image.

Cloud resources are identified by normalized type plus native ID. Cloud resources have no global identifier today. The schema does not invent a synthetic cross-CSP ID, because doing so would require either a registry (operationally expensive and politically fraught) or a hashing scheme (loses the round-trip to the CSP's own tooling). Both options are worse than the alternative. Each CSP's native shape projects into a small normalized type vocabulary, with the native_id alongside. object_store covers an S3 bucket, an Azure Blob container, a GCS bucket; cdn_distribution covers CloudFront, Azure Front Door, Cloud CDN; function covers Lambda, Azure Functions, Cloud Functions; compute_instance covers EC2, Azure VM, Compute Engine. The type is the join key for portfolio reasoning. The native ID stays specific because anyone querying the CSP's API needs it.

Static artifacts are identified by SHA-256. No PURL ecosystem exists for arbitrary static content. The bytes' hash is a perfectly good global identifier and lets consumers compare bit-equality across systems.

Hardware components are identified by HBOM reference. Per CycloneDX HBOM and CISA's 2023 Hardware Bill of Materials Framework, hardware components reference a CycloneDX HBOM document via <bom-ref>@<bom-uri>. The schema slot is reserved; this static-site implementation does not exercise it because there is no hardware to inventory.

Equally important is what the schema deliberately does not normalize: regions, account or subscription identifiers, tag schemas, IAM principal naming, and policy semantics. These vary across CSPs in ways that have no clean projection. They live in attributes, free-form, opaque to the schema. The discipline of naming what is canonical and what is not is itself a contribution. Past attempts at compliance schemas have over-reached, tried to standardize too much, and failed because the standardization cost exceeded the composability benefit. A small, defensible set of canonical names that handles the join cases is more useful than a comprehensive vocabulary that no one adopts.

From Inventory to Reports: KSI Signal and OSCAL SSP

The KSI signal is one report that rides on the canonical inventory. Its validations[] field carries the OPA policy results. Each result references inventory components by component_refs[], an array of component_id values from the inventory. That field is the join key. A portfolio consumer holding KSI signals from many systems can ask "which validations covered component X across the portfolio" and "which components were covered by validation policy P" by joining on component_refs. The same join works regardless of which CSP produced which signal, because the components are canonical.

The OSCAL Rev 5 SSP is a different report on the same inventory. Its structure follows the NIST OSCAL System Security Plan model: system-characteristics, system-implementation (the components), and control-implementation (per-NIST-800-53-control implemented-requirements). The implementation derives the SSP deterministically from the canonical inventory and the FedRAMP KSI catalog (FRMR.KSI.key-security-indicators.json, which maps each KSI to the NIST controls it claims).

The mapping is direct: for each in-scope KSI, the generator iterates the controls listed in the catalog and emits one OSCAL implemented-requirement per unique control. The SSP's components field is sourced from the canonical inventory's components[]. The implementation evidence (the live KSI signal, the Sigstore bundle, the runtime signal, the documentation file recording the implementation rationale) is referenced via OSCAL links with rel: "evidence". A consumer reading the SSP can fetch any link and verify that the asserted control implementation is real.

The two reports never disagree because they're derived from the same source. The KSI signal's signal_id is recorded as a props.ksi-signal-source on the SSP's metadata. Two consumers comparing the two artifacts can confirm by that field that they refer to the same observation. Mismatch is impossible under normal CI operation; if it ever appears, the divergence itself is auditable evidence of a tampered pipeline.

This is the structural point that the dual-emit demonstrates: FedRAMP 20x and Rev 5 are not parallel deliverables that a CSP must produce twice with consistency-by-discipline. They are two views of the same underlying compliance state, derivable from one canonical source. The transition story between the frameworks is therefore a representation question, not a re-evidence-collection project. A CSP migrating from one to the other can keep the same inventory layer and swap reports above it.

The reverse is also true. A future framework (Rev 6, or a state-level program, or an industry-specific overlay) can ride on the same inventory layer with a new report shape above it. The cost of the new framework is the cost of writing the new report generator, not the cost of re-collecting evidence from the system.

Trust Without Keys: The Sigstore Chain

The deploy-time KSI signal is signed via Sigstore keyless using the GitHub Actions OIDC identity. No signing key is held anywhere: not in a hardware module, not in a secret store, not on a developer's laptop. The pipeline asks GitHub for an OIDC token identifying this specific workflow run, hands it to Sigstore's Fulcio certificate authority, receives a short-lived signing certificate bound to that workflow identity, signs the artifact, and discards the ephemeral key. The signature is recorded in the public Rekor transparency log.

In security terms, this gives a trust chain that verifies all the way to public infrastructure. A consumer wanting to verify the signal runs cosign verify-blob with the published bundle and a regular-expression matching the GitHub Actions workflow identity. The verifier checks three things: that the signature is mathematically valid for the bytes of the signal, that the signing certificate was issued by Fulcio to the named GitHub Actions identity, and that the signature appears in the public Rekor transparency log with a valid inclusion proof. None of these checks depends on a privately-held key or a private trust anchor; Fulcio, the OIDC identity, and the Rekor log are all public and independently auditable.

The implications for a portfolio consumer aggregating signals across CSPs are significant. There is no per-CSP key to manage, no per-CSP trust relationship to establish, no rotation schedule to track, no key compromise to recover from. The trust chain reduces to one question: do you trust GitHub's OIDC issuer, Fulcio's CA, and Rekor's log? Three public, audited services with thousands of independent observers. A compromise of any one of them would be detectable by anyone running independent verification against the public log.

This is a substantially better trust model than what most CSP compliance evidence currently offers, where the trust chain typically reduces to "do you trust this CSP's signing process and the integrity of their internal key management?" The PoC shows that the Sigstore-keyless model is implementable in a deploy pipeline today, with no infrastructure beyond GitHub Actions, no on-going key management, and verification mechanics any consumer can run from a laptop.

Differentiation: Per-Control Status and Origination

Each implemented-requirement in the generated SSP carries differentiated values for two OSCAL properties: implementation status and control origination.

Implementation status uses the OSCAL-standard enum:

  • implemented: fully implemented by this system's code and configuration
  • partial: partially implemented, with the rest noted
  • planned: not implemented but planned (tracked in the POA&M)
  • alternative: addressed via compensating control
  • not-applicable: does not apply to this system, with rationale

Control origination uses the FedRAMP-standard enum:

  • sp-system: implemented in this system's code or configuration
  • sp-corporate: implemented at the operator/organization level (training, IR planning, review cadences)
  • shared: partially inherited from the underlying CSP, partially this system's responsibility
  • inherited: fully inherited from the underlying CSP
  • customer-configured: would be configured by an end customer (N/A here; no end customer)
  • customer-provided: would be provided by an end customer (N/A here)

The generator resolves status and origination per control through three layers:

  1. Per-control overrides for ~30 of the most load-bearing controls (CM-2/3/3.2/4/5/6/7/8, AC-2/3/4/6/17, AU-2/3/12, IA-2/3, IR-4/6/8, RA-5, SC-7/8/13, SI-2/3/4/7, SR-3/11) carry hand-written implementation statements that name specific evidence: configuration files, IAM policies, log paths, signed artifacts.
  2. Family-level fall-throughs (covering AC, AT, AU, CA, CM, CP, IA, IR, MA, MP, PE, PL, PM, PS, RA, SA, SC, SI, SR) handle the remaining controls in each family with statements that name how the family is addressed at the operator or inherited level.
  3. The *-1 (policy and procedures) controls in every family route to a templated statement pointing at the relevant documentation.

Live-signal validation results downgrade implemented to partial automatically. If a contributing KSI shows failing validations in the runtime signal, the SSP control downgrades and a remark points at the failure.

The current generator emits 192 implemented-requirements with this distribution: 49% implemented, 34% partial, 17% not-applicable. By origination: 62% sp-system, 21% shared with AWS, 17% sp-corporate, ~1% inherited.

An assessor reviewing this SSP has specific things to verify per control rather than blanket statements to reconcile. The work of assessment shifts from verifying that the SSP is accurate against the live system, which is mechanical, toward verifying that the implementation statements are sufficient, which is judgment.

What This Proves

The implementation proves a set of technical and architectural claims end-to-end. Each item below is observable in the running system, not asserted in the abstract.

Proven.

  • The schema is implementable. The live KSI signal at /.well-known/ksi-signal.json validates against the schema on every deploy.
  • Sigstore keyless signing works in a real deploy pipeline. cosign verify-blob against the published bundle returns success and identifies the issuing GitHub Actions workflow.
  • OSCAL SSP generation is deterministic. Running it twice on the same canonical inventory produces byte-identical output.
  • The runtime emitter and the deploy-time gate evaluate the same compiled Rego. The Lambda loads policy.wasm built from policies.rego at deploy time, so drift between deploy-time and runtime policy is structurally impossible. The runtime emitter reliably detects drift between the policy and the live AWS state by re-evaluating the live configuration daily.
  • The canonical-inventory layer makes reports compose. The same inventory composes into four distinct reports (FedRAMP 20x KSI signal, NIST OSCAL Rev 5 SSP, NIST OSCAL POA&M, and FedRAMP Integrated Inventory Workbook) on every deploy. The composition is not an architectural argument; it is the deploy chain.
  • The differentiated-status SSP is generatable without manual analyst work for the controls covered by family defaults, which is most of them. Of 331 control requirements in the live SSP, about 30 carry hand-written overrides; the remaining ~300 are auto-generated from family defaults and the per-family policy templates. The ratio is observable in the published artifact.
  • Per-family -1 policy and procedures are satisfied by short, system-specific documents in docs/policies/. The OSCAL SSP -1 implementation statements reference these files by path, so an assessor can trace any policy claim to its source document in one click.
  • Public publication of compliance artifacts is viable at this scale. Every artifact this system produces — KSI signal, Sigstore bundle, schema, OSCAL SSP, runtime signal, VDR report — is published at /.well-known/ with no access gating, and the threat model in docs/architecture-decisions.md shows the bounded blast radius of doing so. Strict access control of compliance state is therefore an editorial choice, not a security requirement at this scale. A real regulated CSP would still gate per-customer signals; the architecture supports either mode along the same publication-policy axis.
  • The build is the report for vulnerability detection and response. The VDR aggregator runs in CI on every deploy, classifies findings on the FedRAMP 20x PAIN scale at the Class C tier, blocks the deploy if any finding exceeds Class C tolerance, and emits /.well-known/vdr-report.json. Per-deploy emission satisfies the at-least-monthly cadence in VDR-RPT-PER and VDR-TFR-MHR by overshooting it.
  • The FedRAMP Integrated Inventory Workbook (SSP Appendix M) is derivable from the canonical inventory. The IIW projector reads the published KSI signal and emits an IIW-shaped CSV with the FedRAMP-template column headers. The signal’s components carry the IIW-mandatory fields (function, diagram label, public/private flag, baseline configuration, asset type) as attribute keys; the signal’s top-level ownership block carries the system-owner and application-owner fields. The OSCAL SSP itself already carries the structured inventory in system-implementation/components; the IIW CSV is the same data in the Excel-friendly column shape FedRAMP processes still ask for.
  • The OSCAL POA&M is also derivable from the canonical inventory. scripts/build-oscal-poam.py emits a NIST-OSCAL POA&M document that imports the SSP, identifies POA&M items by FedRAMP-namespaced props, and validates against the official OSCAL POA&M schema. Markdown POA&M (docs/poam.md) is the human-readable view; the OSCAL JSON is the machine-readable view; both reflect the same set of items.
  • Significant Change Notifications are gated at PR time. Every pull request must declare an SCN-Type tag (default routine-recurring); a separate workflow validates the tag. The git history is the SCN audit record (SCN-CSO-MAR).
  • The annual cost of operating this pipeline is roughly $125, observed.

Demonstrated by schema construction, gated on multi-party adoption. The implementation is a single producer (N=1). Several claims the paper makes are about consumption of conformant signals from independent parties — cross-CSP join, portfolio-level aggregation across heterogeneous systems, programmatic third-party-risk consumption of a supplier's published signal. The schema makes each of these correct in principle: PURLs are global by construction, normalized cloud-resource types preserve the cross-CSP join, content hashes are physics, the consumer query is a join on component_refs. A reader inspecting the schema can verify the joins. What is gated on multi-party adoption is the working federation: actual signals from multiple producers, conforming to the same schema, available to consume. The traditional third-party-risk relationship with the underlying CSP is exercised here (the SSP cites AWS authorization package AGENCYAMAZONEW for inherited controls; the SR policy and threat model factor in AWS-side risk explicitly); the schema-driven consumer flow specifically is gated on suppliers publishing conformant signals, which is the same chicken-and-egg pattern every coordination protocol begins with.

Out of scope. Multi-tenant isolation, customer data segregation, regional failover, the specific operational concerns of a CSP at scale, the political economy of adopting a new schema across the FedRAMP marketplace. The PoC is mechanically correct; the institutional work to make this the standard is a different kind of work entirely.

Alignment with FedRAMP’s Announced Direction (NTC-0009)

FedRAMP issued notice NTC-0009 describing initial outcomes from RFC-0024 on machine-readable authorization packages for Rev 5 certifications. Three of the notice’s items map directly onto choices this PoC has already made:

  • “DOCX and XLSX will be retired as an acceptable format in favor of simple text-based equivalents” for Class A–C certifications. This implementation has no DOCX or XLSX in scope. Every artifact is JSON, markdown, CSV, or YAML — text-based and diff-able. The IIW projector emits CSV (the text-based equivalent of the XLSX workbook); the OSCAL SSP carries the structured inventory; the policy documents are markdown. The notice is the direction; this PoC is already past the threshold.
  • “Class D (High) certifications must provide comprehensive machine-readable authorization data” and Classes A–C need “some machine-readable authorization data... along with semi-structured text format similar to the current approach.” This system targets Class C and provides full machine-readable artifacts (KSI signal, OSCAL SSP, OSCAL POA&M, VDR report) plus text-based equivalents (markdown POA&M, family policies, Continuous Monitoring Plan, PTA, Rules of Behavior, recovery plan, IR plan). It overshoots Class C by being machine-readable across the board.
  • “FedRAMP will not require diagrams or illustrations after the transition to the Minimum Assessment Scope.” The boundary diagram in this codebase is now a supplementary human-readable artifact, not a load-bearing compliance deliverable. The MAS rule (already integrated — security-category and information-flow per component) carries the structured-data version of the same information.

The notice also names five Balance Improvement Releases “folding into default requirements”: Minimum Assessment Scope, Significant Change Notifications, Collaborative Continuous Monitoring, Vulnerability Detection and Response, and Authorization Data Sharing. This implementation integrates all five (MAS in the canonical inventory schema; SCN at the PR gate; CCM in the Continuous Monitoring Plan and the runtime KSI emitter; VDR at build time as the published vdr-report.json; ADS by publishing every artifact at /.well-known/ with no access gating).

The argument the paper makes is therefore not “this is what FedRAMP could accept” but “this is the direction FedRAMP itself just announced it will require.” The PoC is a forward-aligned implementation of the published direction.

Business Problems This Addresses

The technical pattern, generalized, addresses several recurring business problems in cloud compliance.

Assessment cost. Most of a 3PAO assessment's cost goes to reconciling the SSP's claims against the live system's actual configuration and writing the gap into the assessment report, not to evaluating control implementations. When SSPs are auto-generated from canonical inventories, the gap is closed by construction: the SSP describes what is. An assessor's time shifts from "verifying that the SSP is accurate" to "verifying that the inventory reflects reality and the implementation statements are sufficient." The first is mechanical; the second is judgment, which is the higher-value activity. The cost of full assessment drops, and the cost of continuous assessment, which is what cATO promises and what 20x is structured around, becomes tractable instead of theoretical.

SSP drift. The most common audit finding across federal cloud authorizations is some form of "the SSP says one thing, the system does another." This finding exists because SSPs are written once, by humans, and the system changes daily. Auto-derivation eliminates the class of finding. Drift cannot exist between the SSP and the system because the SSP is generated from the system on every deploy. A consumer reading the SSP at any moment is reading the current system, not a snapshot from the last formal review.

Framework transition. CSPs in 2026 face a transition between Rev 5 and 20x, and many face simultaneous compliance with state-level programs (StateRAMP, TX-RAMP), industry-specific frameworks (HIPAA, PCI), and customer-driven assertions (SOC 2). Each framework asks for largely the same evidence in a different shape. The dual-emit pattern shows that the cost of supporting multiple frameworks is not multiplicative in the system's complexity; it is additive in the number of report generators above one shared inventory. The work of writing each report generator is bounded and one-time. The work of maintaining each report's content is zero, because the content derives from the live inventory.

Cyber insurance underwriting. The cyber insurance market has struggled to price risk because insurers cannot get reliable, current data on the systems they are insuring. They rely on questionnaires (subject to misrepresentation), point-in-time audits (immediately stale), or self-reported attestations (varying in quality). A canonical inventory with a Sigstore-signed verifiable provenance chain gives insurers something they currently cannot get: a continuously updated, cryptographically attested, machine-queryable description of the insured system's components and their security posture. Whether insurers actually adopt this is a market question, not a technical one. The technical barrier to adoption, "we cannot get reliable data," is structurally addressable.

Vendor risk management. Most third-party-risk-management functions spend the majority of their time chasing answers from suppliers. The supplier sends a SOC 2 report twelve months out of date; the TPRM team translates it into their internal scoring system; the score informs a procurement decision; and by the time the decision is made, the supplier's actual posture has changed. A supplier publishing a continuously updated KSI signal at a stable URL (even if the URL is gated behind authorized customer access) collapses the cycle. The TPRM consumer queries the signal, joins it against their portfolio, and gets an answer in seconds rather than weeks.

The Societal Problem

The patterns above are domain-specific. The structural problem they each address is the same, and it shows up across society wherever risk has to be aggregated across heterogeneous systems run by independent parties.

Insurance markets price risk in aggregate. Banking regulators monitor systemic stability across institutions. Defense planners assess capability across alliances. Public-health authorities track disease across jurisdictions. Each of these depends on being able to compose information across organizations that have no other reason to share data with each other. When the composition is impossible (when every reporting entity uses different identifiers, different schemas, different attestation chains), the aggregator's job becomes structurally bounded by reconciliation cost. Decisions that should be data-driven become judgment-driven, not because judgment is wrong but because the data is unavailable in queryable form.

The canonical-inventory pattern is one instance of a more general structural answer: do the standardization at the layer below where the differentiation actually matters. Reporting frameworks can vary because frameworks express different concerns (compliance, financial disclosure, operational risk, audit). Identifiers cannot vary, because if they vary, no aggregation across reports is possible. The TCP/IP analog is exact here: addresses are standardized; what runs on those addresses is not. Standardizing addresses required a one-time political coordination effort that took years and was repeatedly thought impossible. Once it succeeded, the entire internet became possible on top of it. The application protocols never had to converge.

For compliance and risk management, the equivalent move would be a small set of identifier conventions adopted across regulators, industry consortia, and standards bodies. The components are already there: PURL is well-established in the OSS supply-chain world. SLSA provenance is gaining adoption. Sigstore is operational. CycloneDX HBOM exists. The piece that does not exist is the will to assemble these into a coherent layer, plus the political agreement that this layer is the right place for the standardization to live. The technical work is largely done; the institutional work has barely started.

The claim is not "this PoC will solve insurance pricing or systemic financial risk or supply-chain compromise." Those problems have many causes, and missing identifier standardization is one. The claim is that the missing piece is structurally addressable, and that the addressing happens at a layer that none of the existing standardization efforts in compliance have targeted. Most existing efforts standardize the report (OSCAL, NIST SP 800-53 control catalog, FedRAMP KSI catalog) or the policy (Rego, Sentinel) or the build attestation (in-toto, SLSA) without standardizing the names of the things the reports talk about. The names are the gap, and the names are what compose.

The structural disincentives to adopting this pattern are real. Cloud Service Providers compete partly on inventory opacity; if their customers can compare their offerings on a uniform basis, the comparison is less favorable to the providers with weaker postures. Compliance gatekeepers (auditors, consultants, the industry of producing and reconciling deliverables) earn the gap that canonical inventories would close. Regulators have sunk-cost reasons to maintain current frameworks rather than coalesce around a shared substrate that makes their frameworks interchangeable. Each is a rational local incentive that pulls against the global optimum. The pattern is technically simple. The political economy is exactly as hard as standardizing TCP/IP was.

The PoC contributes an existence proof, not a solution to the political economy: the technical pattern is implementable today, by one person, in spare time, on a static personal website, for $125 a year. Whether it scales to a regulated CSP at federal scale is an open question, but the technical mechanics generalize. Whether enough actors converge on the pattern for it to become the substrate for compliance reporting is also open. The blockers are coordination, not engineering.

Coordination questions are slow and not always solved. TCP/IP took roughly a decade from initial deployment to clear dominance over its competitors. Container shipping took 30 years. The metric system took 200 and is still incomplete. Coordinative protocols win because of patient work plus lucky political openings plus a critical mass of early adopters who tolerate the cost of going first. They almost never win on being technically obviously correct from the beginning.

What Would Change at Scale

The reference implementation answers "is this technically possible?" with a working artifact. The question "what would it take to do this at the scale of a real FedRAMP-authorized CSP?" deserves a direct answer.

Volume of components. A real IaaS CSP has tens of thousands of components in its canonical inventory: every EC2 instance, every Lambda, every container image, every npm package across multiple service teams. The schema scales to that volume because each component is a flat record; the join semantics scale because the joins are by identifier, not by traversal. The bottleneck is the inventory generator, which would need to be incremental rather than rebuilding the inventory from scratch on every deploy. This is solvable engineering, not a structural problem.

Volume of validations. A real KSI signal at scale would carry hundreds of validation results per system per emission. The schema accommodates this; the storage cost is trivial; the consumer-side join performance becomes the question. The right architecture is a portfolio-level aggregator that ingests signals from many systems and exposes a query layer over them. The aggregator can be CSP-internal, agency-internal, third-party, or all three.

Multi-tenancy. A SaaS or PaaS supports many customers, and many of the canonical inventory's components are customer-specific (their data, their configurations). The schema supports this through scoping at the system level (each tenant is a system with its own canonical inventory), but the aggregation pattern needs explicit handling for the customer-responsibility-matrix concerns. A SaaS publishing one canonical inventory for the platform plus per-tenant inventories for each customer's configured surface is the obvious shape; the operational discipline of producing those reliably at scale is not.

Per-customer overlays. Real CSPs face requests from agencies for specific overlays (Agency A wants FIPS 140-3, Agency B wants NIST 800-171, Agency C wants StateRAMP-equivalent). The canonical inventory layer is unaffected; each overlay is another report generator above the layer. The cost of supporting a new overlay is the cost of writing its generator once.

Publication policy. The PoC publishes everything at /.well-known/ because a static personal site has nothing of consequence to hide. A real regulated CSP would publish the schema and the verification mechanism but gate per-system signals through agency-authenticated access. The pipeline is identical; the publication policy is the difference. The architecture supports both modes without changing what gets generated, only what gets exposed where.

The political economy of adoption. The hardest part is not on this list: the institutional work of getting CSPs, agencies, regulators, and standards bodies to converge on a small set of canonical naming conventions across an industry that does not have strong incentives to converge. That work is years of standards-body engagement, FedRAMP engagement, OSS-community engagement, and patient demonstration that early adopters benefit. The PoC is one demonstration; many more would be needed.

Conclusion

FedRAMP 20x is the right paradigm shift. The premise that automated KSI validation makes a separate inventory deliverable unnecessary is right, but it requires that the KSI signals themselves carry standardized component references that compose across CSPs. Without that layer, 20x risks becoming a simulacrum of continuous assurance: the validation runs continuously but the results do not compose into the portfolio view an Authorizing Official actually needs.

The reference implementation on this site shows the layer is buildable. A canonical inventory of the system's components, with PURL for software, normalized cloud type plus native ID for infrastructure, content hash for static, HBOM for hardware, is the primitive that makes everything above it composable. KSI signals ride on that layer. OSCAL Rev 5 SSPs ride on the same layer. SBOMs, vulnerability reports, license inventories, configuration drift detectors all could ride on the same layer. The frameworks above can vary; the substrate underneath is shared.

The implementation is small. The patterns it demonstrates are not. They generalize to cross-CSP risk aggregation, to cyber insurance underwriting, to vendor risk management, to any domain where heterogeneous reports from independent parties have to compose for higher-order reasoning. The technical pattern is simple. The institutional adoption is the hard part, and the institutional adoption is the work that remains.

The work that remains is also the work that mattered for TCP/IP, for double-entry bookkeeping, for the metric system, for container shipping. Coordinative protocols win because of patient work plus lucky political openings plus a critical mass of early adopters who tolerate the cost of going first. They do not win because they are technically obviously correct, even when they are. This implementation is one early adopter, going first on a static personal website, demonstrating that the cost of compliance composability is roughly $125 per year and a few weekends of engineering.

Whether enough other actors follow is a question about coordination, not about the schema. If they do, the structural barrier to portfolio-level risk reasoning across federal cloud, and beyond it, comes down. If they do not, the barrier remains, and the systems that were going to be unsafe at scale because their pieces could not compose remain unsafe at scale. The technical answer exists. The question is whether the institutional choice is made to use it.
