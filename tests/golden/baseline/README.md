# Golden baseline — pre-refactor gate behavior (FROZEN)

Snapshot of how `infrastructure/policies.rego` behaved at commit `4ebf01b`
(pre-refactor), evaluated with the then-pinned **OPA v0.57.0**. These files are
a historical record for diffing the refactor against; they are **not**
regenerated by any Makefile target and must not be edited. Post-refactor
behavior lives in `tests/golden/current/`.

## How each file was produced

- `plan-*.json` — for each raw-plan fixture in `tests/fixtures/plans/`:
  1. `flattener` / `resources[]`: the plan was run through the Python
     flattener extracted **verbatim** from the heredoc in
     `scripts/terraform-plan.sh` (the pre-refactor harness), then each
     flattened resource was evaluated one-at-a-time as
     `opa eval -d policies.rego -i '{"resource": ...}'
     "data.terraform.compliance.compliance_report"` — exactly how the
     pre-refactor gate drove OPA. `flattener: "error: ..."` records that the
     harness crashed on that input (e.g. the empty/garbage fixtures raise
     `KeyError: 'planned_values'`).
  2. `raw_direct_eval`: the same raw plan piped straight into `opa eval`,
     which is what the pre-refactor `README.md` debug instructions said to do.
     Every one of these returns `compliant: true` with zero violations —
     the vacuous fail-open pass the refactor eliminates.
- `html-*.json` — each HTML fixture evaluated as
  `{"html_content": ..., "file_name": ...}`, as the pre-refactor
  `terraform-plan.sh` did per file.

## Known-broken behaviors this baseline intentionally preserves

These are the pre-refactor defects the baseline documents; each is an
**expected delta** against post-refactor goldens:

1. `html-missing-alt.json`: `compliant: true` — the missing-alt rule used an
   RE2-unsupported lookahead regex and could never fire (silent false pass).
2. `plan-*.json` → `raw_direct_eval`: always `compliant: true` — raw plan
   input never matched the bespoke `input.resource` contract (fail-open).
3. `plan-delete-only.json`: the flattener emits the to-be-deleted resource
   list as empty because it reads `planned_values` (correct outcome, but only
   by accident of which section it read).
4. `plan-s3-computed-bucket-name.json`: the name-substring join can never
   link a sub-resource to a bucket whose name is unknown until apply, so the
   bucket is flagged non-compliant on all three synthesized fields.
5. `plan-data-source-website-bucket.json`: all sub-resources pass and the
   (data-source) bucket itself is never evaluated at deploy time — the
   production blind spot covered only by the runtime Lambda.
